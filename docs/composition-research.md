# 作曲エンジンの仮説・検証ログ

作曲エンジンへ新しい理論やルールを追加する前に、次の正本を読むこと。

- [`src/core/music/composition-research.ts`](../src/core/music/composition-research.ts)

このファイルには、比較実験の履歴、暫定結果、未検証の仮説、次に行うA/B比較が構造化データとして保存されている。Web UIの「作曲仮説と検証履歴」も同じデータを直接表示する。

## 更新手順

1. 実装前に `COMPOSITION_HYPOTHESES` へ反証可能な仮説と比較方法を追加する。
2. 比較を実装したら、対応する仮説を `partiallyTested` または `tested` にする。
3. 結果を `COMPOSITION_EXPERIMENTS` へ追加し、仮説の `experimentIds` から参照する。
4. 少数試行や複合条件など、結果からまだ言えないことを `limitations` に残す。
5. 一曲固有の模倣ではなく、複数曲へ一般化できる仮説を優先する。

過去の結果を書き換えて消さず、新しい実験として追記する。結論が変わった場合も、どの実験によって更新されたかを参照で辿れる状態に保つ。

## 検証パイプライン（聴かずに済ませる層）

聴取は最も希少な資源なので、下の層で済むことを耳へ回さない。

0. **一括ファズ監査（CI・`npm run test:all`）** — [`tests/music-fuzz.slow.test.ts`](../tests/music-fuzz.slow.test.ts) が
   全進行×2調性×戦略・シード回転の168曲で、不変条件（クラッシュ・検証エラー・退化・声部交差）ゼロ、
   診断警告のカテゴリ別バジェット、表現レンジのプロファイル（[`src/core/music/audit.ts`](../src/core/music/audit.ts)）を守る。
   新機能の発火・警告削減の効果は、このプロファイルとバジェットの数値変化として聴かずに確認する。
   初回監査の結果は研究ログの EXP-002 を参照。
1. **回帰検出（無料）** — [`tests/music-snapshot.test.ts`](../tests/music-snapshot.test.ts) が
   compose() の聴こえる層（パート×セクションのダイジェスト、[`src/core/music/piece-diff.ts`](../src/core/music/piece-diff.ts)）を
   スナップショット比較する。リファクタでここが落ちたら意図しない音の変化。意図した変更のときだけ、
   失敗差分でどの区間のどのパートが動いたかを確認し、その区間を試聴してから `npm test -- -u` で更新する。
2. **差分の定位（秒）** — ブラインド比較UIは候補間で音イベントが異なる区間を表示し、
   「▶ 差分だけ」でその区間だけ試聴できる（試聴済みには数えない。投票は全体を聴いてから）。
   参考: current vs memoryArc の差分は設計どおり B・D・E の旋律のみ。premiseArc はドラムがAから異なる
   （＝条件3は冒頭から構造識別できる。ブラインドの解釈時に注意）。
3. **票の保全（教師データ）** — 投票には比較を完全再現できる素材（共通 ComposeOptions と匿名化順）が
   埋め込まれる。票は localStorage にしか無いため、UIの「票をエクスポート」で JSON へ定期保全すること。
   蓄積した票は、将来「どの計算可能な特徴が好みを予測するか」を較正する教師データになる。
   インポートで別ブラウザ・別人の票も統合できる（重複は自動除去）。
