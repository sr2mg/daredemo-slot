# 00. 概要とアーキテクチャ

## コンセプト

**daredemo_slot** — 日本のパチスロの内部構造（抽選・リール制御・状態機械）を、WebUI で仕様を設定して保存するだけで「遊べる機種」として動かせるシミュレータ。

- パチスロの数学・制御を学べる教材であり、オリジナル機種を設計して遊べるおもちゃでもある
- ユーザーは A タイプ／AT 機／ストック機などをプリセットから選ぶか、構成要素を自分で組み合わせて機種を定義する

## 非目標（やらないこと）

- 実在機種の再現。実機の ROM データ・抽選テーブル・リール配列・図柄・サウンドは**一切使用しない**
- 換金・課金・賭博性のある要素。プレイマネーのみ
- 実機に対する攻略・解析ツール機能
- **メイン管理型 AT**（5.5 号機以降相当。AT 状態がメインの抽選テーブルを切り替える構造、有利区間・リミッタ）。本設計の不変条件「ナビ層はコアに干渉できない」と両立しないため対象外とする。将来必要になれば「コア側の汎用カウンタ + テーブル切替契機」（RT の一般化）として検討する
- MVP スコープ外: サウンド、凝った演出、モバイル最適化、i18n

## 用語集

| 用語 | 定義 |
|---|---|
| **役** | `roles` / `bonuses` に定義されるエントリの総称。リプレイ・払い出し役・ボーナス役を含む |
| **小役** | 役のうち払い出しがあるもの（リプレイ・ボーナス役を除く） |
| **フラグ** | 抽選テーブルの 1 エントリ = **役 ID の集合**。重複当選は複数役を含む集合で表す |
| **完全打ち / 適当打ち** | 機械割実測の 2 戦略。完全打ち（`perfect`）= 全ナビ通り・目押し完璧、適当打ち（`naive`）= ランダム押し。表記はこの 2 語に統一する |

## 設計原理：メイン基板／サブ基板の分離

実機のアーキテクチャをそのまま設計原理として採用する。

| レイヤー | 実機での対応 | 責務 | 出玉への関与 |
|---|---|---|---|
| **コアエンジン** | メイン基板 | 内部抽選、リール制御、払い出し、役物作動、RT、持ち越し・放出制御 | **これのみが関与** |
| **ナビ層** | サブ基板 | AT/ART の状態管理、押し順ナビの発行 | 情報開示のみ（※） |
| **プレゼンテーション** | 液晶・ランプ・音 | 演出、リール描画、UI | なし |

※ ナビは「メインに元々存在する打ち分け小役の正解」を開示するだけであり、抽選結果を変更しない。技術介入（目押し・JAC ハズシ）と AT ナビは「情報/技術の有無で機械割が変わる」という同型の構造として統一的に扱う。

**不変条件**: コアエンジンはナビ層・プレゼンテーション層の存在を知らない。コアエンジンだけで（ナビなし＝適当打ちプレイヤー相当の）シミュレーションが完結する。

**接続方法**: コアの 1 ゲーム処理は純関数 `(機種定義, エンジン状態, 操作列, RNG) → (新状態, GameEvent)`。`GameEvent` は成立フラグ・入賞役・払い出し枚数・役物作動/終了・RT 移行を含む読み取り専用イベントで、これをナビ層・プレゼン層へ配るのは**コアの外のオーケストレータ（ゲームループ）**の責務。コア自身はナビ層を呼ばない。

## 機種タイプの扱い

A タイプ／B タイプ／C タイプ／AT／ART／ストック機を enum として持つことはしない。タイプは以下の 5 軸の**組み合わせの結果**であり、既知のタイプ名は**プリセット**として提供する（詳細: [01-machine-model.md](01-machine-model.md)）。

1. 役物構成（BB / RB / SB / 集中）
2. 小役構成（払い出し・引き込み保証・打ち分け）
3. RT（リプレイ確率状態と移行契機）
4. 持ち越し・放出制御（ストック・ゼロボ）
5. ナビ（サブ側の AT 状態機械）

## コンポーネントと依存関係

```
[WebUI: 機種エディタ] --(機種定義 JSON を保存)--> [機種定義ストア]
                                                      |
[WebUI: プレイヤー画面] <--(状態/出目/ナビ)-- [コアエンジン] <--(読込)
                                                      |
[配列メーカー] --(生成した配列を検証)--> [コアエンジン(オラクル利用)]
```

- **コアエンジン**: 純関数群 + 明示的な状態オブジェクト。乱数はシード指定可能（リプレイ・テスト・統計検証のため）
- **機種定義**: 単一の JSON（スキーマ: [05-config-schema.md](05-config-schema.md)）。「保存ボタンを押したら遊べる」の実体はこの JSON の書き出しとロード
- **配列メーカー**: 制約を与えるとリール配列を自動生成する。検証にはコアエンジンのリール制御を総当たりオラクルとして使う（[04-reel-layout.md](04-reel-layout.md)）

## 実装順序

1. 抽選エンジン（[02-lottery.md](02-lottery.md)）— 成果物: `EngineState` の型骨格 + テーブル解決規則 + `drawLottery`。テーブル解決規則（02）を確定させてから着手する
2. リール制御 + 手組み配列 1 セットで最小プレイアブル（[03-reel-control.md](03-reel-control.md)）— メモ化と総当たり性質テストを含む
3. 状態機械（ボーナス・RT・持ち越し）、オーケストレータ、ナビ層（[01-machine-model.md](01-machine-model.md)）
4. WebUI（エディタ + プレイヤー）
5. 配列メーカー（[04-reel-layout.md](04-reel-layout.md)）
