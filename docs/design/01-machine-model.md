# 01. 機種構成モデル

機種は 5 つの軸の組み合わせで定義する。A/B/C タイプ等の呼称は軸の組み合わせに付けたプリセット名であり、エンジンはタイプ名を参照しない。

## 軸 1: 役物構成（メイン）

「役物」= 通常時より有利な小役確率状態を作動させる装置。実機の型式区分（A/B/C）の実体はここ。

| 役物 | 内容 | パラメータ |
|---|---|---|
| **BB**（役物連続作動装置） | 高確率の小役ゲームが連続する。いわゆるビッグボーナス | 終了条件（ゲーム数 / JAC 作動回数 / 払い出し上限の組み合わせ）、BB 中抽選テーブル（`tableRef`）、JAC 定義（下記） |
| **RB / JAC**（第一種特別役物） | 短い高確率小役ゲーム。いわゆるレギュラーボーナス | 終了条件（例: 12 ゲーム or 8 回入賞）、作動中テーブル（`tableRef`） |
| **SB**（普通役物） | 1 ゲームだけの小役高確率。単独ではほぼ出玉なし | 作動ゲームのテーブル（`tableRef`） |
| **集中** | 特定テーブルが超高確率になる**並行状態**。パンク役を引くまで継続 | 突入契機、パンク役、集中中テーブル（`tableRef`） |

- **JAC の定義**: BB 内のサブ状態。`{ triggerRole（JAC IN 役の参照）, tableRef（JAC 中テーブル）, end（1 回あたりの終了条件）}`。BB 中は「小役ゲーム ⇔（JAC IN 入賞で）JAC ゲーム」を往復し、JAC 作動回数が BB の終了条件にカウントされる
- **技術介入（ハズシ）**: 専用機能ではない。JAC IN 役を `pullIn: missable`（目押しで外せる配置）にし、BB 終了条件をゲーム数/JAC 回数管理にすると、「JAC を外して小役ゲームを長く回す」打ち方が自然に成立する。**払い出し上限のみで終了する BB ではハズシは原理的に無意味**（総獲得がキャップされるため。実機でも 5 号機の枚数管理 BB 移行でハズシは消滅した）。この組み合わせには保存時に警告を出す
- 歴史的な A/B タイプの違いは JAC 作動回数の規定差（A=3 回、B=2 回以下）。B タイプが「出玉が少ない」わけではない（BB 中が小役ゲーム主体になるぶん、小役設計次第で大量獲得型にもなる）。本エンジンではどちらも BB のパラメータ差
- **集中の注記**: 実機では小役の集中（〜3 号機）と SB 集中（4 号機 C タイプ）が規則上区別されるが、本エンジンでは「特定テーブルが超高確率になる、パンク役まで継続する並行状態」として統一的に扱う（意図的な単純化）
- 役物を一切搭載しない構成も許す（ボーナスレス AT 機の表現）

## 軸 2: 小役構成（メイン）

各役（リプレイ・払い出し役・ボーナス役を含む。用語は [00](00-overview.md) の用語集参照）は以下の属性を持つ。

| 属性 | 説明 |
|---|---|
| `payout` | 払い出し枚数（リプレイは再遊技、ボーナス役は 0 で作動契機） |
| `pullIn` | `guaranteed`（PB=1: どこを押しても引き込める配置を配列側に要求）/ `missable`（目押し要素。目標引き込み率を配列側に要求） |
| `nav` | 打ち分け指定。`group`（所属 navGroup）、`correct`（正解押し順 or 正解狙い位置）、`onMiss`（不正解時: 取りこぼし `lose` / 別役入賞 `reduced` + 参照役） |

- **navGroup** はトップレベルで定義するグループ。同グループに属する複数の役（例: 押し順ベル 3 択なら 3 つの役）がそれぞれ自分の正解を宣言する。**抽選テーブルのフラグは役 ID の集合**なので、3 択は 3 エントリになる（[02](02-lottery.md)）
- **MVP の実装範囲**: 正解判定は**第 1 停止リールのみ（`correctFirst`、3 択）**。第 1 停止で有効役が確定するため、リール制御の決定論・メモ化がそのまま保てる（初期 AT 機の実態にも近い）。完全な押し順 6 択・狙い位置型（目押し型）は将来拡張
- **同一図柄組み合わせの共有**: 押し順ベル 3 択は同じ `pattern`（ベル揃い）を共有する別フラグ。入賞判定・蹴飛ばしは役 ID ではなく**図柄組み合わせ単位**で行う（[03](03-reel-control.md)）
- **リプレイの打ち分け**（維持リプ/転落リプ）も同じ仕組みで表現する。ただしリプレイは規則上必ず入賞するため、リプレイ役の `onMiss` は `reduced`（別のリプレイ役を参照）のみ許可し、バリデーションで強制する（[03](03-reel-control.md) 優先度 2 参照）
- **打ち分け小役の存在が AT の成立要件**である。ナビ（軸 5）は navGroup の正解を開示することでしか出玉に影響できない。打ち分けなしの機種にナビを載せても何も起きない（設計上のバグではなく仕様。WebUI は警告を出す）

## 軸 3: RT（メイン）

RT = リプレイ確率が変動した状態。**基底テーブルのリプレイ系エントリの重みだけを差し替えるオーバーレイ**として表現する（テーブル解決規則: [02](02-lottery.md)）。フルテーブルの二重管理はしない。

- RT 状態のリスト。各状態は `replayWeights`（リプレイ役 ID → 重み）を持つ
- **移行契機**（`entry` / `exit` の `on`）: `bonusEnd`（役物終了）/ `roleHit`（特定役入賞: 昇格・転落リプレイ、転落小役）/ `games`（規定ゲーム数消化）/ `bonusFlag`（ボーナスフラグ成立 = 内部中 RT）
- ART = 「AT（軸 5）が、リプレイ高確率 RT の維持に必要な打ち分け正解（転落リプ回避の押し順）をナビする」構成。エンジン上の特別扱いではなく、移行契機と navGroup の設定で自然に成立する

## 軸 4: 持ち越しと放出制御（メイン）

`carryover` セクションとして常設する（ストックの有無で構造は変えない）。

- **フラグ持ち越し**: ボーナス系フラグは入賞するまで持ち越す（内部中）。小役・リプレイは単ゲームで消滅。これは全機種共通の基本規則
- **キュー**: 持ち越しは FIFO キュー。`queueLimit: 1` が単純持ち越し（通常機）、`> 1` がストック機。内部中に新たなボーナスが当選した場合の扱いは [02](02-lottery.md) の規則 3 参照（キュー空きがあれば積む、上限超過分は破棄）
- **蓋（放出制御）**: 「内部中だがボーナスを入賞させない」状態。実機では高リプレイ RT ＋ボーナスを揃えられない制御で実現されるが、エンジンでは `lid: on/off` として単純化する。`on` の間はリール制御がキュー先頭のボーナスを非成立扱いで蹴飛ばす（[03](03-reel-control.md) 優先度 3）
  - `engageOn`（蓋を掛ける契機）: `bonusFlag`（**ストックが空→非空になったとき**。放出ゾーン中の上乗せでは掛け直さない）/ `bonusEnd`（ボーナス終了時に掛け直し = 連チャン管理）
  - `release`（解除 = 放出の契機）: `gameCountTable`（解除ゲーム数の重み付き抽選。0G を含めれば即連）/ `lottery`（解除抽選。`on: any` = 毎ゲーム / `on: pureMiss` = **純ハズレ**（抽選結果が空集合）のゲームのみ。サラ金型 KC の契機）/ `roleHit`（特定役で解除）
  - **モード**（`modes`）: 解除テーブルの選択状態（通常/天国など）。ボーナス終了時に重み付き抽選でモード移行し、掛け直しは**新モードの**解除テーブルで行う。出玉に直結するのでメイン管理（吉宗型の 1G 連 = 天国モードの解除テーブルに 1G を厚く盛る）
  - **ポップ規則**: 放出されたボーナスが入賞するとキューから除去。キューに次のフラグが残っていれば直ちに内部中に戻り、`engageOn` に従って蓋が再作動する。`engageOn: [bonusFlag]` のみの機種では、蓋が開いたあとキューが尽きるまで毎ゲーム放出が続く（= 放出ゾーン。サラ金型 KC）
- **ゼロボ**: 払い出し設計がほぼゼロの BB/RB。出玉ではなく状態遷移のトリガー（RT の張り替え、`engageOn: bonusEnd` による蓋の掛け直し）として使う。専用機能は不要で、上記の組み合わせで表現できる

## 軸 5: ナビ（サブ）

サブ基板に相当する層。**メインの抽選・制御には一切干渉できない**。できることは 2 つだけ:

1. コアが 1 ゲームごとに発行する **GameEvent**（成立フラグ、入賞役、払い出し枚数、役物作動/終了、RT 移行を含む読み取り専用イベント。[00](00-overview.md) 参照）を購読する
2. プレイヤーに正解押し順／狙い位置を**表示**する（対象: `navTargets` に列挙した navGroup）

その上で AT 状態機械を持つ:

- **AT 抽選契機**（`triggers.on`）: `roleHit`（特定役成立）/ `gamesCeiling`（ゲーム数天井）/ `duringBonus`（役物作動中の抽選）。抽選はナビ層専用の独立乱数で行う（メイン乱数とは別系統: [02](02-lottery.md)）
- **継続管理**（`management.type`）: `games`（ゲーム数）/ `set`（セット数 × 1 セットのゲーム数 + 継続率）/ `stock`（ナビ層独自のセットストック。軸 4 のフラグキューとは別物）/ `payoutDiff`（差枚数）
- **上乗せ**（`addOn`）: AT 中の特定役で継続量を加算

**スコープ制限**: AT 状態がメインの抽選テーブルを切り替える「メイン管理型 AT」（5.5 号機以降・有利区間相当）は本シミュレータの対象外とする（[00](00-overview.md) 非目標）。ナビ層はあくまで情報開示のみ。

## プリセット（タイプ名 → 軸の組み合わせ）

| プリセット | 軸1 役物 | 軸2 打ち分け | 軸3 RT | 軸4 蓋 | 軸5 ナビ | モデルの雰囲気 |
|---|---|---|---|---|---|---|
| **A タイプ** | BB(JAC3)+RB | なし | なし | なし | なし | ジャグラー系 |
| **A タイプ（技術介入）** | BB(JAC3・ゲーム数管理・JAC IN が missable) | 目押し打ち分け | なし | なし | なし | 大花火系 |
| **B タイプ** | BB(JAC2 以下・小役ゲーム主体) | なし | なし | なし | なし | 4 号機 B（大量獲得型も可） |
| **C タイプ** | SB+集中 | なし | なし | なし | なし | 集中機 |
| **AT 機** | BB+RB（薄め）or 役物なし | 押し順 6 択 | なし | なし | AT | 獣王系 |
| **ART 機** | BB+RB | 押し順（ベル+リプレイ） | 高確 RT + 転落リプ | なし | AT（RT 維持ナビ） | 5 号機 ART |
| **ストック機** | BB+RB | なし | 内部中 RT | あり（モード付き解除） | なし | 吉宗系 |
| **SB ストック機** | SB(+BB) | なし | なし | あり（純ハズレ解除・放出ゾーン） | なし | サラ金系 |

プリセットは機種定義 JSON の初期値を埋めるだけのテンプレートであり、選択後に全軸を自由に編集できる。「C タイプ + AT」のような実機に存在しない合成も動くこと（それがシミュレータの面白さ）。

## ゲームフロー状態機械

コアエンジンの 1 ゲームは**純関数** `(機種定義, エンジン状態, 操作列, RNG) → (新状態, GameEvent)` として処理する。ナビ層・プレゼン層への通知はコアの外のオーケストレータ（ゲームループ）が GameEvent を配ることで行う（[00](00-overview.md)）。

処理順:

```
ベット（クレジットから投入 / リプレイ時は自動再ベット）
→ レバー ON → 内部抽選（02）→ リール回転
→ 停止操作 ×3（リール制御 03）→ 入賞判定 → 払い出し
→ 状態遷移（役物作動/終了、RT 移行、キュー・蓋更新）→ GameEvent 発行
```

### 状態の構造

エンジン状態は「基本状態 × 修飾状態」の直積として持つ:

- **基本状態**（排他）: `通常` / `役物作動中（どの役物か + 役物内サブ状態: BB の小役ゲーム/JAC ゲーム）`
- **修飾状態**（それぞれ独立に on/off・値を持つ）: `RT 状態`（軸 3）/ `内部中`（キュー非空）/ `蓋`（軸 4）/ `集中`（軸 1）

直積のどの組み合わせでどの抽選テーブルが使われるかは、[02](02-lottery.md) の解決規則が一意に定める。

```
            ボーナスフラグ成立            入賞
  [通常/RT] ────────────→ [内部中(蓋on/off)] ──→ [役物作動中]
      ↑                                              │ 終了条件
      │              （キューに残があれば内部中へ戻る）  │
      └──────────── ボーナス終了時 RT へ ←────────────┘
  [集中] は並行の修飾状態（パンク役で解除）
```
