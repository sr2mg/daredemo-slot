# 02. 内部抽選エンジン

## 基本方式

実機に倣い、**レバー ON の瞬間に 1 回だけ乱数を引き、そのゲームの成立フラグを確定する**。リール停止操作は結果を変えられない（制御が変わるだけ）。

- 乱数範囲: 0〜65535（16bit）。実機の慣習に合わせることで「1/240 ≒ 273/65536」のような実機風の仕様表記がそのまま使える
- 乱数源: シード指定可能な決定論的 PRNG（xoshiro / PCG 等）。同一シード + 同一操作列 → 同一結果を保証（テスト・リプレイ・統計検証のため）
- **メイン乱数とナビ層乱数は別系統**（ナビ層の AT 抽選がメインの出目再現性を壊さないため。セーブデータには両方の状態を保存する: [05](05-config-schema.md)）

## フラグの表現

**フラグ = 役 ID の集合**として構造化する。文字列の命名規則（`cherry+bb_red` 等）には依存しない。

```jsonc
// 抽選テーブル = エントリの配列。重み合計 ≤ 65536、残余は「ハズレ」（空集合）
[
  { "roles": ["replay_keep"], "weight": 8978 },       // ≒1/7.3
  { "roles": ["bell_LCR"],    "weight": 1092 },        // 押し順6択は6エントリ
  { "roles": ["cherry"],      "weight": 1057 },
  { "roles": ["cherry", "bb_red"], "weight": 66 },     // 重複当選 = 複数役の集合
  { "roles": ["bb_red"],      "weight": 200 },
  { "roles": ["rb"],          "weight": 273 }
]
```

- **1 ゲーム 1 エントリ**: 抽選結果はテーブルの 1 エントリ。エンジンが暗黙にフラグを合成することはない。重複当選はエントリ側に集合として明示する
- テーブル内の役 ID は機種定義の `roles` / `bonuses` への参照であり、保存時バリデーションで整合を検証する

## テーブル解決規則（基底 + オーバーレイ）

エンジン状態は基本状態 × 修飾状態の直積（[01](01-machine-model.md)）なので、テーブルは状態ごとの一枚岩ではなく**合成**で解決する。優先順に:

1. **役物作動中**: 作動中の役物（BB 小役ゲーム / JAC / RB / SB）の `tableRef` が指すテーブルを**丸ごと**使用。修飾状態は無視する（最優先）
2. **集中中**（役物非作動時）: 集中の `tableRef` のテーブルを丸ごと使用
3. **それ以外**: `基底テーブル` に修飾を重ねる:
   - **設定修飾**: `settingOverrides[設定値]` でエントリの重みを差し替える（基底 = 設定 1）
   - **RT 修飾**: 現在の RT 状態の `replayWeights` で、リプレイ系エントリの重みを差し替える（重みの増減分はハズレと相殺）
   - **内部中修飾**: ボーナス役を含むエントリの扱いを持ち越し設定で分岐:
     - **キューに空きがない**（`queueLimit` 到達 or ストックなし機で持ち越し中）: ボーナス役を含むエントリはハズレに置換（当該エントリに小役が同居する場合は小役のみ残す）
     - **キューに空きがある**（ストック機）: ボーナス当選は**キューに積む**。当該ゲームの入賞制御には乗せない（制御に乗るのは常にキュー先頭のみ。蓋 on ならそれも蹴飛ばし対象: [03](03-reel-control.md)）

状態名とテーブルの対応:

| エンジン状態 | 使われるテーブル |
|---|---|
| 通常 | `lottery.base` |
| RT 中 | `lottery.base` + `rtStates[].replayWeights` |
| 内部中（通常/RT） | 上記 + 内部中修飾 |
| BB 小役ゲーム | `bonuses[bb].tableRef` |
| JAC ゲーム | `bonuses[bb].jac.tableRef` |
| RB 中 / SB 作動 | `bonuses[rb|sb].tableRef` |
| 集中中 | `concentration.tableRef` |

`lottery.tables` に置くのは役物・集中から `tableRef` で参照される丸ごとテーブルのみ。予約キーは `base` だけで、暗黙の文字列一致による結合はしない。

## フラグの持ち越し

- ボーナス役のみ、入賞まで持ち越す（詳細は [01](01-machine-model.md) 軸 4 のキュー・蓋・ポップ規則）
- 小役・リプレイはそのゲーム限り
- 当該ゲームの有効フラグ = `キュー先頭のボーナス（蓋 off の場合）+ 当該ゲームの抽選エントリ`。どちらを優先して引き込むかはリール制御の責務（[03](03-reel-control.md)）

## 設定値

設定 1〜6（本数は `lottery.settings` で可変。省略時 1 = 設定なし）。**基底テーブル = 設定 1** とし、設定 2〜6 は `settingOverrides`（roles 集合が一致するエントリの重み上書き。無ければ追加）の**差分オーバーレイ**で表現する。テーブル解決規則では RT 修飾より先に適用する。役物中テーブル（`tables`）は設定共通（MVP の単純化。実機の BB 中設定差は将来拡張）。

設定はエンジン状態（`EngineState.setting`）であり、遊技中は変わらない（ホール側の操作）。

## 機械割の検証

抽選テーブル + 配列 + 制御が揃えば、機械割は**完全打ち**（`perfect`: 全ナビ通り・目押し完璧）と**適当打ち**（`naive`: ランダム押し）の 2 通りをモンテカルロで実測できる（用語は [00](00-overview.md) 用語集に統一）。

- CLI: `simulate --config machine.json --games 10_000_000 --strategy perfect|naive --seed N`
- WebUI の機種エディタは保存時にこのシミュレーションを回し、設定別機械割・ボーナス確率の実測値を仕様表として表示する（「自作機種の雑誌スペック表」が自動生成される体験）
- 完全打ちと適当打ちの機械割差が、そのまま技術介入度・ナビ依存度の指標になる

将来的には、目標機械割から重みを逆算する調整補助（感度分析）を載せるが、MVP では実測表示のみ。

## インターフェース（概略）

```ts
type FlagSet = ReadonlySet<RoleId>;   // 当該ゲームの成立役集合（複合フラグ含む）
type LotteryResult = {
  drawn: FlagSet;                     // 抽選エントリの役集合
  queuedBonus: BonusId | null;        // このゲームでキューに積まれたボーナス（ストック機）
};
function resolveTable(machine: MachineDef, state: EngineState): WeightedEntry[]; // 解決規則の実装
function drawLottery(table: WeightedEntry[], rng: Rng): FlagSet;
```

純関数 + 明示的 RNG。状態遷移（キュー操作・持ち越しの発生・消滅）は入賞判定後の遷移フェーズで行い、抽選関数自体は状態を変更しない。

**実装順序上の注意**（[00](00-overview.md) ステップ 1 の成果物）: `drawLottery` は `EngineState` 型に依存するため、ステップ 1 の完了条件は「`EngineState` の型定義（遷移ロジックなしの骨格）+ テーブル解決規則 + drawLottery」とする。
