# 03. リール制御

## 原理

規則由来の制約: ストップボタン押下から **190ms 以内に停止**しなければならず、リール回転速度には上限（約 80rpm）がある。その帰結として滑りは**最大 4 コマ**（物理的な制約ではなく規則の産物）。したがって停止候補は常に「押下図柄 + 先 4 コマ」の **5 択**。リール制御とは、この 5 択から 1 つを選ぶ決定論的な関数である。

実機にはテーブル方式（事前計算した停止テーブル）とコントロール方式（逐次探索）があるが、本エンジンは**その場で 5 候補を評価するコントロール方式**を採る。理由: 機種定義から自動で正しい制御が導出でき、ユーザーに停止テーブルを書かせずに済む。出目の作り込み（リーチ目等）はデフォルト滑りテーブルというオプション層で行う（後述）。

## 押下位置の量子化（出玉に直結する UI 仕様）

`pushPosition` はプレゼン層が生成するが、これは演出ではなく**出玉に直結するゲームプレイ仕様**なのでここで確定する:

- リール回転は一定速（約 80rpm 相当 = 実機準拠）。フレームレートに依存せず、**押下時刻からコマ番号への変換は決定論的**に行う（回転開始時刻とコマ周期から算出。描画は結果に影響しない）
- 目押し補助（低速回転モード、押下位置表示など）は UI 設定であり機種定義には含めない。補助の有無は機械割実測（perfect/naive）に影響しないよう、シミュレーションは UI を介さずコマ番号を直接与える

## 制御関数

```ts
type StopEvent = { reel: number; pushPosition: number; stopPosition: number };

function resolveStop(
  machine: MachineDef,          // 配列・役・ライン定義
  activeFlags: FlagSet,          // 当該ゲーム成立役 + キュー先頭ボーナス + 蓋状態
  history: readonly StopEvent[], // 停止済みリールの順序付き操作履歴
  reel: number,                  // いま押下されたリール
  pushPosition: number           // 押下位置（コマ番号）
): number;                       // 滑りコマ数 0..4
```

純関数。同じ入力には常に同じ滑りを返す（**この決定論性が配列メーカーの検証オラクル（04）の前提**）。

- `history` は「順序付き」であることが本質: 押し順 6 択の正解判定には第 1〜第 3 停止の**順序**が必要で、停止済み集合だけでは復元できない
- `stopPosition` だけでなく `pushPosition` も保持する: 狙い位置型打ち分け（目押し型）の正解判定は「滑り後にどこに止まったか」ではなく「どこを狙って押したか」で行うため

## 評価ルール（優先度順）

5 候補それぞれについて「その位置で止めた場合の帰結」を評価し、以下の優先度で選ぶ。

1. **【絶対・除外】蹴飛ばし**: 非成立の**図柄組み合わせ**の入賞が確定する候補は選ばない（判定は役 ID ではなく pattern 単位。同一 pattern を共有する別フラグ＝押し順ベル 3 択などは、成立役と同じ組み合わせなら合法）。第 1・第 2 停止では「残りリールでどう止めても非成立役の入賞を回避できなくなる停止形」も先読みして除外する（残りリールの全押下位置に対して回避可能な滑りが存在するか、を再帰的に確認する。計算量は後述のメモ化で抑える）
2. **【絶対・優先】リプレイ**: 成立時は**いずれかのリプレイ役**の入賞を保証する（規則由来。配列制約により常に可能）。複数のリプレイ役が絡む場合（維持リプ/転落リプの押し順リプレイ）に**どの**リプレイを揃えるかは、優先度 5 の navGroup 規則が決める。この分業のため、リプレイ役の `onMiss` は `reduced`（別リプレイ役の入賞）のみ許可し、取りこぼし（`lose`）は保存時バリデーションで拒否する
3. **【蓋】**: 蓋 on の間、キュー先頭のボーナスは非成立扱いで蹴飛ばす（ストック機の放出制御。[01](01-machine-model.md) 軸 4）
4. **【引き込み】成立役を最大限入賞に向かわせる**。複数の役が同時に有効な場合（例: 内部中の小役成立ゲーム）の優先モードは機種定義で選択:
   - `payout-first` 枚数優先（払い出しが大きい役を優先）
   - `count-first` 個数優先（入賞可能性のある役の数を優先）
   - `bonus-first` ボーナス優先 / `role-first` 小役優先
   ※ 小役優先の機種では「ハズレのゲームだけボーナスが引き込まれる」ため、リーチ目（通常あり得ない出目）が制御の副産物として自然発生する
5. **【打ち分け】** navGroup 役は「成立役の正解（MVP: `correctFirst` = 第 1 停止リール）と実際の操作が一致するか」で引き込み対象が変わる。不正解時は役定義の `onMiss` に従い、取りこぼし（蹴飛ばし）または参照役（減額役・別リプレイ）の引き込みを行う。**実装**: 第 1 停止の瞬間に有効役集合を解決して制御コンテキストを差し替える（GameSession）。以降は通常の引き込み・蹴飛ばしと同じ
6. **【デフォルト滑り】**: 上記で候補が複数残った場合の選択規則。既定は「最小滑り」。オプションで `(フラグ, リール, 押下位置) → 優先停止位置` の**出目テーブル**を機種定義に持たせ、リーチ目・チャンス目のデザインを可能にする（省略可。省略しても遊技性は成立する）

## 計算量とメモ化（必須要件）

先読み再帰のコストは見かけより大きい。コマ数 N のリールで第 1 停止の安全性判定を素朴に行うと、
`候補5 × 残り2リールの順序2 × 第2押下N × 候補5 × 第3押下N × 候補5 ≒ 250·N²`（N=20 で約 10 万評価/コール）となり、総当たり検証（全押下 N³ × 押し順 6 × フラグ状態数）に掛け合わせると素朴実装は 10¹⁰ 評価規模になり破綻する。

そこで**メモ化を設計上の必須要件とする**:

- 安全性・引き込み可能性の判定は `(フラグ状態キー, 停止済み各リールの停止位置)` にのみ依存し、操作順序・押下履歴に依存しない部分問題である
- ユニーク部分問題数は `フラグ状態数 × (N+1)³` 程度（N=20 で 1 フラグあたり約 9,300）に過ぎず、メモ化すれば総当たり検証全体が「ユニーク部分問題数 × 部分問題あたり評価コスト」に潰れて現実的になる
- フラグ状態は「当該ゲーム役集合 × キュー先頭ボーナス種 × 蓋 on/off」の直積で数える（当該ゲーム役集合単独ではない）

## 有効ライン

- ライン定義は機種定義の一部（例: 中段 1 ライン、クロス 5 ライン、変則ライン）
- MVP は「3 枚掛け・5 ライン（水平 3 + 対角 2）」をデフォルトとする
- 入賞判定は「停止後、いずれかの有効ライン上に役の図柄組み合わせが揃っているか」の単純照合

## 押し順の扱い

- 停止操作は 6 通りの順序を任意に受け付ける（変則押し自由）
- navGroup（`type: order`）の正解判定は `history` の順序で行う
- navGroup（`type: position`）の正解判定は対象リールの `pushPosition` が正解範囲に入っているかで行う

## 技術介入（JAC ハズシ等）の表現

技術介入は制御の特別機能ではなく、既存要素の組み合わせで自然に表現される（[01](01-machine-model.md) 軸 1 の JAC 定義参照）:

- JAC IN 役を `pullIn: missable` にし、BB をゲーム数/JAC 回数管理にすると「JAC を外して小役ゲームを引き延ばす」打ち方が成立する
- 完全打ちと適当打ちの機械割差（[02](02-lottery.md)）が、そのまま技術介入度の指標になる

## 縮退規則（合法候補ゼロの場合）

制御の正しさは配列が検証済みであることを前提にする。**保存時パイプライン（[05](05-config-schema.md)）を通過していない機種定義はプレイヤーにロードできない**、を仕様とする（手組み・手動微調整した配列も必ず検証を通る）。したがって `resolveStop` で合法候補がゼロになることは運用上あり得ず、発生した場合は例外を投げる（エンジンのバグか未検証データの混入を示す）。

## テストと検証

- **性質テスト**: (a) 非成立役が入賞するケースが存在しない (b) 成立リプレイの取りこぼしが存在しない (c) 蓋 on 中にボーナスが入賞しない (d) PB=1 役の取りこぼしが存在しない — を全フラグ状態 × 全押下位置 × 全押し順の総当たり（メモ化前提）で検証する
- この総当たり検証器はエンジンのテストであると同時に、配列メーカー（[04](04-reel-layout.md)）の受け入れ判定器そのものである
