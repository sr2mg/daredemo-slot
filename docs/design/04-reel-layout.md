# 04. リール配列 — 制約・検証・自動生成

## 方針

リール配列（各リールの図柄並び）は、機種の要件から**決定論的に自動生成できる**。ユーザーは図柄を 1 コマずつ並べる必要はなく、役ごとの性質（PB=1 / 目押し要素 / ボーナス図柄数）を宣言するだけでよい。手組みも引き続き可能（自動生成はオプション）。

## 表現

- リールは図柄 ID の循環配列。コマ数は機種定義で指定（既定 20、21 も可）
- 3 リール構成を既定とする（4 リール以上は将来拡張）

## 制約の定義

役の属性（[01](01-machine-model.md) 軸 2）から、配列への制約が機械的に導出される。

| 役の宣言 | 導出される配列制約 |
|---|---|
| `pullIn: guaranteed`（PB=1） | 各リールで当該役の構成図柄が**最大間隔 5 コマ以内**（押下点含む 5 択でどこからでも引き込める）。コマ数 N なら最低 ⌈N/5⌉ 個配置（N=20 で 4 個、N=21 で 5 個） |
| `pullIn: missable`, 目標引き込み率 r | 構成図柄の間隔分布から計算される引き込み率（ランダム押下時に 5 択に入る確率）が r±許容誤差に収まる配置 |
| ボーナス役 | 構成図柄（例: 赤 7）を各リール 1〜2 個。引き込み率は自動的に低くなる（目押し要素） |
| navGroup（押し順打ち分け） | 正解/不正解で異なる図柄組み合わせが必要 → グループ内の各役に別図柄（または同図柄の別ライン）を割り当てられること |
| 見た目制約（任意） | 同一図柄の連続数上限、特定図柄の隣接禁止など。遊技性に影響しない審美的制約 |

さらに、**配列単体では検証できない大域制約**が 2 つある:

1. **蹴飛ばし可能性**: どのフラグ状態・どの押し順・どの押下位置でも、非成立役を回避できる候補が 5 択に必ず残ること
2. **リプレイ完全引き込み**: 制御を通した実測で取りこぼしゼロであること

これらは配列とリール制御の相互作用で決まるため、**リール制御エンジン（03）を検証オラクルとして総当たりで確認する**。ケース数は 押下位置 N³ × 押し順 6 × フラグ状態数（当該ゲーム役集合 × 持ち越しボーナス種 × 蓋 on/off）で数百万規模だが、各ケース内の先読み再帰が重いため**メモ化が必須**（部分問題の共有により「ユニーク部分問題数 × 評価コスト」に潰れる。詳細: [03](03-reel-control.md) の計算量節）。

## 生成アルゴリズム

制約充足問題として解く。MVP は単純な**ランダム再スタート + 局所探索（焼きなまし）**で十分:

```
1. 役の宣言から必要図柄数を算出し、初期配列をランダム生成
   （PB=1 役は「間隔 ≤5」を満たすよう先に骨組み配置してから残りを埋める）
2. コスト関数 = Σ 制約違反（局所制約は直接計算、大域制約はオラクル総当たりの違反数）
3. 図柄のスワップ/差し替えを近傍操作として焼きなまし
4. コスト 0 で採用。規定イテレーションで到達しなければ図柄数を調整して再スタート
```

- 生成は決定論的にする（シード付き）。同じ機種定義 + 同じシード → 同じ配列
- 大域制約のオラクル評価は重いので、局所制約を満たした候補にのみ実行する 2 段階評価にする。オラクル側のメモ化（[03](03-reel-control.md)）はフラグ状態キー単位なので、配列が変わるたびにキャッシュは破棄する（配列自体もキーに含めれば近傍解間で部分共有も可能だが、まずは破棄で十分）

## 出力と可視化

- 生成結果は機種定義 JSON の `reels.strips` に書き込まれる（生成後は手動微調整可能）
- WebUI は配列を縦帯で可視化し、役ごとの実測引き込み率・取りこぼし率を注記する（例: 「チェリー: 引き込み率 48.2%」）

## 既定プリセット配列

自動生成の実装前でも遊べるよう、手組みの配列セットを 1 つ以上同梱する（A タイププリセット用）。これは 03 の制御エンジン開発時のフィクスチャを兼ねる。
